# ЛОКАЛЬНЫЕ СЕТИ И ТЕСТИРОВАНИЕ ПО: СИСТЕМАТИЧЕСКИЙ КОНСПЕКТ

## 1. Локальные вычислительные сети (ЛВС / LAN)

**ЛВС (LAN — Local Area Network)** — компьютерная сеть, объединяющая вычислительную технику и периферийное оборудование (принтеры, серверы, хранилища) на ограниченной территории: в пределах одного здания, офиса, кампуса или группы помещений.

**Ключевые характеристики ЛВС:**
- высокая скорость обмена данными (от 100 Мбит/с до 100 Гбит/с)
- совместный доступ к ресурсам (принтеры, файлы, приложения)
- низкая задержка передачи
- управляемая среда (администрирование принадлежит организации)

**Физическая реализация:**
- проводные технологии: Ethernet (витая пара — Cat5e/6/6a, оптоволокно)
- беспроводные технологии: Wi-Fi (IEEE 802.11 a/b/g/n/ac/ax)

**Архитектурная модель ЛВС:**
- коммутируемая звезда (star) — доминирующая топология
- центральный элемент — коммутатор (switch)
- возможна иерархия коммутаторов (access → distribution → core)

> *Примечание: иллюстративный материал к разделу предполагает наличие схемы ЛВС с обозначением рабочих станций, сервера, коммутатора и периферийных устройств.*

## 2. Жизненный цикл (ЖЦ) компьютерных систем

**Жизненный цикл (ЖЦ)** — последовательность стадий от зарождения идеи до вывода системы из эксплуатации.

**Основные стадии ЖЦ:**
1. Анализ требований (requirements analysis)
2. Проектирование (design)
3. Реализация / кодирование (implementation / coding)
4. Тестирование (testing)
5. Внедрение (deployment)
6. Эксплуатация и сопровождение (operation & maintenance)
7. Утилизация / вывод из эксплуатации (retirement)

**Модели ЖЦ:**

| Модель | Принцип | Применение |
|--------|---------|------------|
| Каскадная (waterfall) | последовательное выполнение этапов без возвратов | проекты с фиксированными, понятными требованиями |
| Итеративная (iterative) | цикличное повторение этапов с постепенным уточнением | крупные системы, требования которых эволюционируют |
| Спиральная (spiral) | управление рисками, каждый виток — прототип | высокорисковые проекты, R&D |

> Цель внедрения моделей ЖЦ: повышение эффективности разработки, снижение проектных и технических рисков.

## 3. Структурированные кабельные системы (СКС)

**СКС (Structured Cabling System)** — единая физическая инфраструктура здания или комплекса зданий, построенная по иерархическому принципу.

**Объединяемые подсистемы:**
- локальные вычислительные сети (ЛВС / LAN)
- телефонные сети (голосовая связь)
- системы видеонаблюдения (CCTV)
- системы безопасности и контроля доступа (ACS)

**Свойства СКС:**
- стандартизированная передача данных (соответствие ISO/IEC 11801, TIA/EIA-568)
- гибкость переключений (patch-панели, коммутационные шнуры)
- высокая надежность (резервирование, экранирование)
- масштабируемость (возможность наращивания без замены магистралей)
- унифицированные компоненты (коннекторы, кабели, розетки)

**Типовая иерархия СКС:**
- рабочая область (work area) — розетка → рабочая станция
- горизонтальная подсистема (horizontal) — этажное распределение
- магистральная подсистема (backbone) — вертикальные каналы между этажами/зданиями
- аппаратная / серверная (equipment room)

## 4. Тестирование ПО: базовые понятия

**Тестирование** — процесс проверки продукта на соответствие требованиям, выявление дефектов (багов), оценка качества, надежности и удобства. Включает не только выполнение программы, но и планирование, анализ результатов и обеспечение качества в целом.

**Интеграционное тестирование** — этап, следующий за модульным (unit-тестированием). Отдельные компоненты или модули объединяются и проверяются на совместимость и корректность взаимодействия.

**Цели интеграционного тестирования:**
- выявление дефектов в обмене данными между модулями
- обнаружение несовместимости интерфейсов
- поиск ошибок логики взаимодействия функционально завершенных частей

---

## 5. Методологии тестирования: «ящики»

Три подхода, различающиеся уровнем доступа к внутреннему устройству системы.

| Метод | Доступ к внутренностям | Позиция | Что проверяет |
|-------|------------------------|---------|----------------|
| Чёрный ящик (black box) | нет | пользователь | внешние функции, соответствие спецификациям |
| Белый ящик (white box) | полный | разработчик | внутренняя логика, пути выполнения кода |
| Серый ящик (gray box) | частичный | тестировщик с архитектурными знаниями | функциональность с учётом структуры |

**Характеристики серого ящика:**
- источники знаний: документация, структура БД, API-спецификации, логи
- цель: интеграционное тестирование, поиск уязвимостей, сокращение времени
- примеры: тестирование API (знание формата запроса/ответа), проверка БД при действиях в интерфейсе

> Комбинирование всех трёх подходов обеспечивает максимальное покрытие.

## 6. Фаззинг (Fuzzing)

**Фаззинг** — автоматизированный метод тестирования, при котором программе подаются случайные, некорректные или неожиданные входные данные для провокации сбоев.

**Область выявления:**
- переполнение буфера (buffer overflow)
- зависания (hang / deadlock)
- утечки памяти
- недекларированное поведение

**Принцип работы:**
1. Фаззер генерирует множество входных данных («корпус» / corpus)
2. Данные подаются на вход целевой программе
3. Система наблюдает за реакцией (крах, аномалия, штатное поведение)

**Виды фаззинга:**

| Тип | Принцип | Применение |
|-----|---------|------------|
| Генерационный (generation-based) | данные создаются с нуля на основе понимания формата/протокола | форматы файлов, сетевые протоколы |
| Мутационный (mutation-based / gray-box) | изменение существующих корректных данных | быстрое покрытие кода, популярен в индустрии |

**Инструменты:** AFL (American Fuzzy Lop), libFuzzer, Honggfuzz.

**Область обязательного применения:**
- разработка защищенного ПО
- критическая информационная инфраструктура (КИИ)

> Фаззинг-тесты стабильнее обычных: они проверяют инварианты, а не конкретные тест-кейсы, и не требуют частой поддержки при изменении кода.

## 7. Отладка и отладчики (Debuggers)

**Отладчик (debugger)** — инструмент (встроенный в IDE или самостоятельный), предназначенный для поиска, анализа и исправления ошибок в программном коде.

**Основные возможности:**
| Функция | Описание |
|---------|----------|
| Точки останова (breakpoints) | приостановка выполнения на заданной строке |
| Пошаговое выполнение (stepping) | построчное выполнение с контролем состояния |
| Просмотр переменных (introspection) | значения переменных, объектов, структур в текущей точке |
| Стек вызовов (call stack) | последовательность функций, приведшая к текущей точке |
| Изменение переменных «на лету» | модификация данных в памяти во время остановки |

**Типовые среды с отладчиками:**
- PyCharm, VS Code, IntelliJ IDEA
- 1С:Предприятие
- IBM Rational Application Developer
- GDB (GNU Debugger) — консольный

> Отладчик незаменим для выявления логических ошибок, утечек памяти и анализа поведения кода в рантайме.

## 8. Тест-кейс (Test Case)

**Тест-кейс** — детализированный сценарий (набор шагов, условий, входных данных) для проверки конкретной функции программного продукта.

**Структура тест-кейса:**

| Элемент | Содержание |
|---------|-------------|
| Уникальный идентификатор | ID тест-кейса |
| Цель / название | что именно проверяется |
| Предварительные условия (preconditions) | состояние системы до начала теста |
| Шаги (steps) | пошаговая инструкция действий |
| Входные данные | конкретные значения (логины, файлы, параметры) |
| Ожидаемый результат (expected result) | эталонное поведение системы |
| Фактический результат (actual result) | заполняется при выполнении |

**Функции тест-кейсов:**
- систематизация проверок (покрытие всех требований)
- воспроизводимость (любой исполнитель получает тот же результат)
- основа для автоматизации (автотесты пишутся по тест-кейсам)

> Тест-кейс позволяет объективно отличить корректную работу программы от бага.

## 9. Альфа- и бета-тестирование

| Тип | Исполнители | Среда | Цель |
|-----|-------------|-------|------|
| Альфа-тестирование (alpha) | внутренние специалисты (QA, разработчики) | изолированная / тестовая | поиск критических ошибок на ранней стадии |
| Бета-тестирование (beta) | реальные пользователи (внешняя группа) | «боевые» условия | сбор отзывов, выявление мелких недочетов |

**Иерархия:**
Альфа → (исправление критических дефектов) → Бета → (сбор обратной связи) → Релиз

## 10. Регрессионное тестирование (Regression Testing)

**Регрессионное тестирование** — проверка ПО после внесения изменений (исправлений багов, добавления новой функциональности, рефакторинга) для подтверждения, что эти изменения не нарушили уже работающие функции.

**Что проверяется:**
- отсутствие побочных эффектов (regression bugs)
- сохранение общей стабильности
- неизменное качество ранее работавших модулей

**Особенности:**
- включает повторное выполнение ранее пройденных тест-кейсов
- критически важно в циклических методологиях (Scrum, Agile)
- часто автоматизируется для ускорения
- выполняется перед каждым релизом / спринтом

> Без регрессионного тестирования каждое изменение с высокой вероятностью ломает существующую функциональность.
